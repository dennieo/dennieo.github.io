# Автоматический торговый бот для Binance — архитектура и план реализации

*Версия 1.0 — рабочий документ (paper)*

---

## 1. Цель и рамки проекта

Построить автономного торгового бота, который:

- торгует на **спотовом рынке Binance** через официальный API (без плеча и фьючерсов — для старта с малым балансом это принципиально: убыток ограничен вложенной суммой);
- работает **полностью автоматически**: сам получает данные, анализирует их, принимает решения, выставляет ордера и ведёт учёт;
- имеет **три режима работы** с одинаковым кодом стратегии:
  1. `paper` — симуляция на реальных рыночных данных, деньги не используются вообще;
  2. `testnet` — торговля на официальном тестнете Binance (виртуальный баланс, настоящий API);
  3. `live` — реальная торговля небольшим балансом;
- защищён жёстким **риск-менеджментом**: лимит риска на сделку, лимит дневного убытка, аварийный стоп (kill switch).

**Почему Binance.** Самая большая ликвидность, лучшая документация API, официальный тестнет (https://testnet.binance.vision), низкие комиссии (0.1%, ниже с BNB). Мы используем библиотеку **ccxt**, поэтому смена биржи (Bybit, OKX, Kraken) — это одна строка в конфиге, а не переписывание кода.

**Честное предупреждение.** Никакая стратегия не гарантирует прибыль; большинство простых ботов на долгом горизонте проигрывают комиссиям и рынку. Цель первого этапа — не заработать, а построить надёжную инфраструктуру и *проверить стратегию данными*. Торгуйте только суммой, потерю которой готовы принять полностью.

## 2. Общая архитектура

Бот — это один процесс с бесконечным циклом, который просыпается по расписанию (по закрытию свечи), прогоняет конвейер «данные → анализ → решение → исполнение → учёт» и засыпает. Никаких очередей и микросервисов на старте — при балансе в сотни долларов это лишняя сложность.

```
                 ┌──────────────────────────────────────────────┐
                 │                  main loop                    │
                 │        (тик по закрытию свечи, poll)          │
                 └──────┬───────────────────────────────────────┘
                        ▼
   ┌────────────┐   ┌────────────┐   ┌────────────┐   ┌────────────┐
   │  Exchange  │──▶│ Indicators │──▶│  Strategy  │──▶│    Risk    │
   │  (ccxt)    │   │ EMA/RSI/ATR│   │  сигналы   │   │ размер поз.│
   │ OHLCV/цены │   └────────────┘   │ BUY/SELL/  │   │ лимиты,    │
   └────────────┘                    │ HOLD       │   │ kill switch│
                                     └────────────┘   └─────┬──────┘
                                                            ▼
   ┌────────────┐   ┌────────────┐                    ┌────────────┐
   │  Notifier  │◀──│   State    │◀───────────────────│  Executor  │
   │ (Telegram) │   │  (SQLite)  │                    │ ордера:    │
   └────────────┘   │ позиции,   │                    │ paper/     │
                    │ сделки,    │                    │ testnet/   │
                    │ equity     │                    │ live       │
                    └────────────┘                    └────────────┘
```

### Модули

| Модуль | Ответственность |
|---|---|
| `exchange.py` | Единая обёртка над ccxt: свечи, цены, балансы, ордера; переключение testnet/live; повторы при сетевых ошибках |
| `indicators.py` | Чистые функции расчёта EMA, RSI, ATR на pandas — используются и ботом, и бэктестером |
| `strategy.py` | Превращает свечи в сигнал `BUY / SELL / HOLD`. Ничего не знает про деньги и ордера |
| `risk.py` | Считает размер позиции от риска, проверяет лимиты (макс. позиций, дневной убыток, мин. ордер) |
| `executor.py` | Исполняет решения: в `paper` — симулирует сделку с комиссией, в `testnet`/`live` — рыночные ордера через API |
| `state.py` | SQLite: открытые позиции, история сделок, кривая капитала. Бот переживает перезапуск без потери состояния |
| `notifier.py` | Уведомления в Telegram о сделках и ошибках (опционально) |
| `main.py` | Цикл: оркестрация всех модулей, обработка ошибок, логирование |
| `backtest/` | Прогон той же стратегии по истории: доходность, просадка, winrate |

Ключевой принцип: **стратегия и риск — чистые функции без побочных эффектов**. Один и тот же код исполняется в бэктесте, на бумаге, на тестнете и вживую. Это исключает классическую ошибку «в бэктесте одна логика, в проде другая».

## 3. Алгоритм анализа и принятия решений

### 3.1 Данные

- Источник: REST-эндпоинт свечей (OHLCV) Binance через ccxt, таймфрейм по умолчанию **1h**.
- Анализируем только **закрытые** свечи — текущая недоформированная свеча отбрасывается, иначе сигналы будут «мигать» (repainting).
- На старте берём ~300 свечей для разогрева индикаторов, дальше обновляем хвост.

### 3.2 Стратегия v1: трендовая (EMA-кроссовер + RSI-фильтр + ATR-стопы)

Простая, объяснимая, с понятными точками отказа. Только лонг (спот).

**Вход (BUY)** — все условия одновременно:
1. EMA(12) пересекла EMA(26) снизу вверх на последней закрытой свече — начало восходящего импульса;
2. RSI(14) < 70 — не покупаем в перегретый рынок;
3. риск-модуль разрешает (лимиты не нарушены, хватает баланса).

**Выход (SELL)** — любое из условий:
1. цена коснулась стоп-лосса: `entry − 2.0 × ATR(14)`;
2. цена коснулась тейк-профита: `entry + 3.0 × ATR(14)`;
3. EMA(12) пересекла EMA(26) сверху вниз — тренд закончился.

ATR (средний истинный диапазон) делает стопы **адаптивными к волатильности**: в спокойном рынке стоп ближе, в бурном — дальше. Соотношение тейка к стопу 1.5:1 означает, что стратегия прибыльна уже при ~45% выигрышных сделок.

Все параметры (периоды, множители) вынесены в `config.yaml` — подбираются бэктестом, не хардкодятся.

### 3.3 Расчёт размера позиции (fixed fractional risk)

Рискуем фиксированной долей капитала на сделку (по умолчанию **1%**):

```
риск_в_$   = equity × 0.01
дист_стопа = 2.0 × ATR                # расстояние до стоп-лосса
кол-во     = риск_в_$ / дист_стопа
кол-во     = min(кол-во, доступный_баланс / цена)   # не больше, чем есть денег
```

Пример: баланс $500, BTC = $60 000, ATR = $900 → стоп на $1800 ниже входа. Риск $5 → покупаем 0.00278 BTC (~$167). Если стоп сработает — теряем ровно $5, т.е. 1% капитала. Десять стопов подряд — минус ~10%, не катастрофа.

### 3.4 Защитные механизмы

| Механизм | Поведение |
|---|---|
| Лимит дневного убытка | Если за сутки потеряно > 5% капитала — бот закрывает позиции и прекращает торговлю до следующего дня |
| Макс. открытых позиций | По умолчанию 2 — нет концентрации риска |
| Мин. размер ордера | Binance не примет ордер < ~$10; проверяем заранее |
| Повторы и деградация | Сетевые ошибки → повтор с паузой; повторные ошибки → пропуск тика, уведомление; бот не падает от одного сбоя API |
| Восстановление | Состояние в SQLite: после перезапуска бот знает свои позиции и продолжает их вести |
| Kill switch | Файл `STOP` в папке бота → немедленная остановка новых входов |

## 4. Поток одного тика (псевдокод)

```
каждые poll_interval секунд:
    если появилась новая закрытая свеча:
        для каждого символа из конфига:
            свечи      = exchange.fetch_ohlcv(символ)
            индикаторы = compute(свечи)               # EMA, RSI, ATR
            позиция    = state.get_position(символ)

            если позиция открыта:
                если цена <= стоп или цена >= тейк или сигнал SELL:
                    executor.close(позиция)           # market sell
                    state.record_trade(...)
                    notifier.send("закрыл X, PnL ...")
            иначе:
                если сигнал BUY и risk.allows():
                    qty = risk.position_size(equity, ATR)
                    executor.open(символ, qty, стоп, тейк)
                    state.save_position(...)
                    notifier.send("купил X по Y")

        state.snapshot_equity()                        # кривая капитала
        risk.check_daily_loss()                        # kill switch
```

Стопы и тейки на первом этапе контролирует сам бот (проверка каждым тиком). На этапе live стоит перейти на **OCO-ордера Binance** — тогда стоп исполнится на стороне биржи, даже если бот упал (см. дорожную карту).

## 5. Безопасность

1. **API-ключи только в переменных окружения** (`.env`, не коммитится — в `.gitignore`).
2. На ключе включаем **только** «Enable Reading» и «Enable Spot Trading». **Никогда** не включаем «Enable Withdrawals» — даже украденный ключ не позволит вывести деньги.
3. Привязка ключа к IP сервера (whitelist в настройках Binance).
4. Отдельный саб-аккаунт/кошелёк под бота с рабочей суммой — основной капитал бот физически не видит.
5. Логи не содержат ключей; уведомления — только факты сделок.

## 6. План внедрения (по этапам)

**Этап 0 — инфраструктура (готово в этом репозитории).**
Скелет бота, все модули, конфиг, бэктестер, документация.

**Этап 1 — бэктест (1–2 недели календарных данных работы).**
Прогнать стратегию на 1–2 годах истории BTC/USDT и ETH/USDT. Смотрим: общую доходность против buy&hold, максимальную просадку, winrate, profit factor, число сделок. Подбираем параметры, но осторожно — переподгонка (overfitting) под историю это способ красиво слить депозит. Правило: параметры выбраны на одном периоде — проверены на другом (out-of-sample).

**Этап 2 — paper trading (2–4 недели).**
Режим `paper`: реальные данные, виртуальный баланс $500. Убеждаемся, что бот стабильно живёт неделями, сигналы совпадают с бэктестом, состояние переживает перезапуски.

**Этап 3 — testnet (1–2 недели).**
Режим `testnet` с ключами от testnet.binance.vision. Проверяем реальное исполнение ордеров: фильтры лотов, минимальные суммы, частичные исполнения, ошибки API.

**Этап 4 — live с малым балансом.**
$100–300, не больше. Первые недели — ежедневный контроль через Telegram-уведомления и кривую капитала. Критерий останова заранее: просадка > 15% → выключаем и анализируем.

**Этап 5 — развитие (дорожная карта).**
- OCO-ордера (стоп на стороне биржи);
- WebSocket-поток вместо REST-опроса (реакция быстрее, лимиты API свободнее);
- вторая стратегия (mean-reversion на боковике) и переключение по режиму рынка;
- метрики в Grafana/просто CSV-отчёты; walk-forward-оптимизация;
- вынос в отдельный репозиторий + деплой на VPS в Docker (Dockerfile уже есть).

## 7. Деплой

Бот должен работать 24/7 → нужен VPS ($4–6/мес: Hetzner, DigitalOcean) или домашний мини-сервер.

```bash
docker build -t trading-bot .
docker run -d --restart unless-stopped --env-file .env \
  -v $(pwd)/data:/app/data trading-bot
```

`--restart unless-stopped` + состояние в SQLite на примонтированном томе = бот сам поднимается после сбоев и перезагрузок сервера.

## 8. Чего этот бот сознательно НЕ делает (v1)

- Не торгует фьючерсами и не использует плечо — при малом балансе это самый быстрый путь к нулю.
- Не использует ML/нейросети — сначала нужна инфраструктура и baseline, который они должны будут обогнать.
- Не занимается HFT/арбитражем — там конкуренция с колокацией и комиссиями, малому балансу не место.
- Не «усредняется» против позиции (martingale/DCA в убыток) — классический способ превратить малый убыток в полный.
